Build the software,
then grow it — on a live dashboard.

agentic-suite is a Claude Code plugin — an SDLC + GROW framework that bundles three orchestration skills (conductor + agentic-app-builder + agentic-worker). The conductor chains BUILD → GROW for a single request needing both software and the business work on top of it (SEO, marketing, sales, research, content): a parallel-agent swarm ships the product, hands off a factual brief, then grows it against what was actually built — all in your Claude Code session, no API key.

Install

Installing the plugin registers its three bundled skills (conductor + agentic-app-builder + agentic-worker) and the shared persona registry.

/plugin marketplace add FaisalNoman/agentic-suite
/plugin install agentic-suite@agentic-suite

Requirements: Claude Code + Node.js 18+ (zero-dependency .mjs helpers — dashboard, gate, doctor). No API key; runs on your subscription, in-session.

Quick start

Just prompt — nothing to set up first. On a fresh run the conductor auto-runs a pre-flight check, then drives the rest.

Build a photo-manager web app, then write its SEO strategy,
a content-marketing plan, and a sales proposal.

The conductor builds the app, gates completion, hands off, produces the growth deliverables against the real product, and opens an interactive showcase of everything.

Build-only

“Build me X” → routes straight to agentic-app-builder.

Grow-only

“Research X and write a report” → routes to agentic-worker.

Mixed

“Build X then market it” → the full BUILD → GROW chain.

Unclear

The conductor asks you once (on the dashboard) which you meant.

Get started

Routing — which engine runs

At Stage 0 the conductor classifies intent. Rule of thumb:

  • Produces or edits runnable code → BUILD participates.
  • Produces a document / dataset / strategy → GROW participates.
  • Both → the BUILD → GROW chain. Neither clear → it asks once.

A marketing-flavoured noun doesn’t force GROW if the deliverable is software — “an SEO tool” is a BUILD; “landing-page copy” is GROW.

PromptRuns
“Build a todo web app” / “fix this bug” / “add dark mode”builder only
“Build a landing page / website”builder only (it’s code)
“Research & compare 5 CRM tools” / “write an SEO strategy”worker only
“Write landing-page copy + 3 launch tweets”worker only
“Build a photo manager, then write its SEO + GTM plan”builder → worker
“Build a tool that generates SEO reports”builder only (it’s a tool)
“Improve my site’s SEO” / “scrape competitor prices”conductor asks (tool vs strategy/data)
Two things people miss. Pre-build research is not GROW — “research the best stack then build it” is builder-only (research feeds builder’s own planning). And order is hard-wired BUILD → GROW (growth depends on the product existing).
Engines

BUILD — agentic-app-builder

The software engine. Three modes auto-detected:

GREENFIELD

Build from scratch: interview → design/wireframe → plan → TDD → parallel impl → review.

FEATURE

Add to an existing codebase from a scoped feature spec.

SURGICAL

Diagnose & fix a bug/error with reproduction-first debugging.

What makes it different

  • Global dependency-graph scheduler — every task whose deps are met runs in parallel (up to a concurrency cap), spanning all features at once. Milestone gates live inside the DAG; no per-sprint barriers.
  • Design routing — picks the best installed design skill (ui-ux-pro-max first) and generates a design system before any UI code.
  • Wireframe/demo approval on the dashboard — builds a throwaway demo/index.html, opens it, and collects Approve / Suggest changes on the board before real code.
  • TDD (tests before impl, enforced as DAG edges) + two-stage review (spec + quality) per milestone.
  • Per-language coding rules injected per code agent (TS/JS/Py/Go/Rust/Java/C#).
  • Model tiering — impl agents on the cheap tier, architect/review on the top tier (real cost control).
Engines

GROW — agentic-worker

The growth engine. Classifies the task domain, decomposes the brief into a task DAG, and runs specialists in parallel on its own scheduler — then merges with a coherence pass.

  • 192-persona specialist registry across 15 domains (research, marketing, sales, strategy, content, finance, …), with scored, scoped routing — only the needed persona loads per task.
  • Bring-your-own-docs gate — under the suite it reads HANDOFF.json and skips re-interviewing, grounding all growth work in the real product.
  • Showcase — merges deliverables into grow/outputs/ and generates an interactive showcase.html (searchable, per-doc TOC, copy-code) that auto-opens when ready.
Engines

Handoff & the completion gate

Between BUILD and GROW the conductor enforces a deterministic, script-checked gate — GROW never starts on a failed or partial build.

  • Stage 2.5 gate — runs check-build-gate.mjs (reads milestones + RESULT.json); on fail it stops and reports the blocked task. Trusts the script’s exit code, not prose.
  • HANDOFF.json — product, features, stack, URLs, decisions, shared memory, and a build_status block — the cold-start brief GROW grows against.
  • Session boundary — for a large build, the conductor can recommend resuming GROW in a fresh session from HANDOFF.json to keep context sharp.
Dashboard

Live dashboard

Each phase opens a real-time, zero-dependency board (BUILD on :4317, GROW on :4318) served over SSE. It’s two-way — questions and approvals happen on the board, not the CLI.

Swarm

Live agent cards with role, status, working / done / blocked, a model-tier badge, and a model-mix summary (predict spend).

Plan

The dependency graph as an animated flowchart — nodes flow ready → working → done. Plan-approval modal with 🗺 Open Plan.

Tests

Per-milestone unit results + live Playwright E2E stream when present.

Replay

Scrub the event log to reconstruct any past moment — auto-populated from status transitions.

Prompts & approvals (interview, plan, wireframe demo) pop as modals on the board via a one-call bridge; the demo opens itself, and you Approve / Suggest changes there. Tokens show real measured spend, split into fresh · cache · out (cache reads are ~10× cheaper, so the headline isn’t misread).

Dashboard

Persona routing

GROW routes each task to a specialist from the shared registry by scoring your task text against persona descriptions (threshold-gated; one persona per task). Preview the pick for any requirement without a run:

node ~/.claude/skills/agentic-suite/scripts/route-preview.mjs \
  "design an SEO strategy and keyword plan for a SaaS" --domains marketing,strategy

At runtime the chosen persona shows as a badge on the agent card.

Flow

Workflow diagram

End-to-end: the conductor pre-flights, classifies, splits, runs BUILD (parallel DAG swarm) to a gated completion, hands off a factual brief, then runs GROW (parallel specialists) and opens the showcase. Each phase drives its own live board.

Flow

Operating principles

The non-negotiables the orchestrator follows on every run.

Mode-first

Detect GREENFIELD / FEATURE / SURGICAL before anything else — the mode decides which stages run.

Parallel fan-out is mandatory

When N tasks are ready, all N agents are dispatched in one message — never one-at-a-time for independent work.

Isolated writes

Each subagent owns exactly one file; the scheduler defers any task whose writes overlap an in-flight task (runtime file-ownership locks).

Interfaces → tests → impl

Ordering is enforced as edges in the DAG (tdd depends on architect; impl on tdd), not as serial phases — so chains hold globally while everything else runs.

Every step is a card

Interview, planner, gate, commit — every unit of work appears as a live dashboard card. No silent steps.

Crash-resume

State is persisted after every phase; on re-run the orchestrator reads it, skips the done set, re-queues interrupted work, and continues.

Flow

Conductor stages

The thin suite-level orchestration that chains the two engines.

0 · pre-flight + classify→ 1 · split→ 2 · BUILD→ 2.5 · gate→ 3 · handoff→ 4 · GROW→ 5 · wrap
  1. Stage 0 — Pre-flight & classify. Auto-run suite-doctor; on a blocking failure, stop. Then classify intent into {needs_build, needs_grow, build_brief, grow_brief} and route (build-only / grow-only / chain / ask).
  2. Stage 1 — Split. Produce two scoped briefs from the one request, confirm once on the dashboard, persist to suite-state.json.
  3. Stage 2 — BUILD. Invoke agentic-app-builder in build/ (board on :4317); let its full pipeline run.
  4. Stage 2.5 — Gate (blocking). Run check-build-gate.mjs; proceed only if every milestone is done and RESULT.json (if present) is done. On fail: stop, report, offer resume.
  5. Stage 3 — Handoff. Synthesize HANDOFF.json (product, features, stack, URLs, decisions, build_status); copy into grow/plan/docs/ so GROW skips re-interviewing.
  6. Stage 4 — GROW. Invoke agentic-worker in grow/ (board on :4318) against the real product.
  7. Stage 5 — Wrap. Combined summary; point at the showcase + both dashboards’ Replay tabs.
Flow

BUILD flow (agentic-app-builder)

What runs inside Stage 2. Mode is detected first; the stage set follows.

GREENFIELD scratch

Full pipeline: interview → design/wireframe → plan → TDD → parallel impl → review → finish.

FEATURE extend

Repo survey → feature scope → impact analysis + scoped plan → TDD → impl → stricter integration gate.

SURGICAL fix

Problem capture → systematic-debugging dispatch → targeted fix → verify. Reproduction-first.

Stage 0 — preflight (all modes)

  1. Resume check — if framework-state.json exists, resume; skip the rest of Stage 0.
  2. Mode detection — directory state + request language → GREENFIELD / FEATURE / SURGICAL.
  3. Git init + capability check — detect git; warn if absent (commits/undo disabled).
  4. Dir setup — plan/state, plan/docs, etc.
  5. Dashboard launch — start the server (--no-open), open the board via the foreground helper.
  6. Worktree decision (G+F) — isolate in a git worktree for large builds.
  7. Foreground vs background + budget cap (G+F).

Stages 1–3 — docs → plan → build loop

  1. Bring-your-own-docs gate — use provided docs, else run the interview (Phases A–D) — all on the dashboard.
  2. Plan + global DAG + doc-quality gate — publish the dag (animated Plan tab), get explicit approval on the board.
  3. Wireframe loop (UI only) — throwaway demo/index.html, Approve / Suggest on the board.
  4. Scaffold the toolchain — working build/test before any TDD agent runs.
  5. One global scheduler run — architect → tdd → impl (parallel) → milestone gate (integration + lint) → two-stage review → commit, all as DAG nodes with model-tiered agents.
  6. Phase 9 — finishing — final verification, cleanup, BUILD-SUMMARY, final commit, worktree merge, completion message; distil lessons.
Flow

GROW flow (agentic-worker)

domain detect→ docs gate (HANDOFF)→ decompose→ specialists ‖→ review gate→ merge→ showcase
  1. Domain detection — classify into research / analysis / content / marketing / sales / strategy / … (drives which specialists + review criteria apply).
  2. Docs gate — under the suite, HANDOFF.json is the product brief; re-interview is skipped.
  3. Decompose — coordinator builds a task DAG (one deliverable per task) with a registry-driven agent + scheduler role per task.
  4. Specialist fan-out — parallel agents, each routed to a scored persona from the 192-registry (scoped loading); model-tiered.
  5. Review gate — high-stakes tasks auto-insert a review node; coherence checked across tasks.
  6. Merge + showcase — combine into grow/outputs/; generate + auto-open showcase.html.
Operate

Pre-flight — suite-doctor

On a fresh run the conductor auto-runs the doctor; it proceeds only if the environment is sound, else stops with exactly what to fix. Run it yourself anytime:

/suite-doctor
# or
node ~/.claude/skills/agentic-suite/scripts/suite-doctor.mjs

Checks node, both skills, the registry, dashboard ports, write access, suite-state.json integrity, git, and whether the optional hooks are installed. PASS / WARN / FAIL with exit 0 / 1 / 2.

Operate

Enforcement hooks (opt-in)

The suite runs hook-free by default. For long or unattended runs, install an opt-in pack that turns prose rules into deterministic guards the whole swarm (incl. subagents) can’t skip. They’re dormant unless a run is active and fail-open (a hook bug never breaks your session).

node ~/.claude/skills/agentic-suite/scripts/install-hooks.mjs   # --scope user for global
# then restart Claude Code (hooks load at session start)
HookEventGuards
config-protectionPreToolUseno weakening test/lint/tsconfig configs mid-run (creation allowed; edits to existing blocked)
protect-statePreToolUseno editing .git / settings / the hook & dashboard infra
dangerous-bashPreToolUseno rm -rf /, git push --force, reset --hard, curl|sh, fork bombs
circuit-breakerPostToolUsetrips after 5 consecutive tool failures (bounded loops)
cost-persistPostToolUsepersists token spend + alerts on a budget breach
state-verifySessionStartresume nudge if a run is mid-flight
precompact-snapshotPreCompactsnapshots state before context compaction

Escape hatches: plan/state/.allow-config-edit, plan/state/.allow-danger. Remove all with uninstall-hooks.mjs.

Operate

Cross-run learning

Each run distils a few atomic lessons from its signals and user overrides (plan changes, wireframe suggestions, interview overrides) into .agentic-builder/lessons.json, deduped and confidence-weighted. The next run warm-starts from the relevant ones — so the suite gets sharper the more you use it.

Operate

State & resume

suite-state.json (conductor) and framework-state.json (each engine) are the source of truth. On a re-run or a fresh session, state is read and the run resumes where it stopped — completed phases skipped, the gate re-checked, GROW resumed from its DAG.

/suite-resume        # deterministic briefing: phase, outstanding milestones, next action
Heads-up. Skill & dashboard-template changes apply to the next fresh run after a Claude Code restart — a running session keeps the old skill in memory, and an in-progress project keeps the template it copied at start.

Architecture

The conductor is deliberately thin: it classifies, sequences, gates, and bridges — it owns no build or growth logic. The two engines do the real work, and a single shared agents/ folder holds the registry both resolve via ../agents/registry.json (one copy, no duplication).

agentic-suite/skills/
  agents/                shared registry (192 personas, 15 domains) + persona .md
  conductor/             the BUILD→GROW orchestrator  ("agentic-suite")
    scripts/  hooks/  commands/   doctor · gate · hooks · scan · resume · route-preview
  agentic-app-builder/   software engine  → resolves ../agents
  agentic-worker/        growth engine    → resolves ../agents
